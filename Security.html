1 Security

1.1 Node.js - security

1.1.1 CSRF protection

app.use(express.csrf());
app.use(function(req, res, next) {
    res.locals.csrftoken = req.csrfToken();
    next();
}

And then somewhere in the HTML: 

<input type="hidden" name="_csrf" value="{{csrftoken}}">
1.1.1.1 Express CSRF middleware ignores verifying tokens on HTTP `GET`, `OPTIONS` and

`HEAD` requests.

1.1.1.2 Ensure `GET` APIs are non-mutating.

1.1.2 Preventing internal implementation disclosure

1.1.2.1 Remove the `X-Powered-By` header.
app.disable("x-powered-by");
1.1.2.2 Change the default session cookie name (from `connect.sid`)
app.use(express.session({
    secret: "13375UP4H4X0R",
    key: "yourCusomtKey",
    /* ... */
});

1.1.3 XSS protection

1.1.3.1 It's a myth that template libraries handle output encoding by default, making the application safe from XSS attacks.
1.1.3.2 Encode untrusted data for correct context, depending on where it will reside.
  • HTML Body
    • & -> `&amp;`
    • < -> `&lt;`
    • > -> `&gt;`
    • " -> `&quot;`
    • ' -> `&#x27;`
    • / -> `&#x2F;`
  • HTML attributes
    • e.g. `<input type="text" name="firstname" value="{untrusted}">`
    • Non-alphanumeric chars -> `&#xHH;` format
  • CSS
    • e.g. `<div style="width={untrusted};">contents</div>`
    • Untrusted ata -> CSS hex encoding (`\HH` or `\HHHHHH`)
  • JS
    • e.g. `<script>var firstName="{untrusted}"</script>`
    • Non-alphanumeric chars -> `\uXXXX;` unicode format
  • URL
    • e.g. `<a href="{untrusted}">Show details</a>`
    • Untrusted data -> `encodeURI()`
  • URL parameter
    • e.g. `<a href="/account?id={untrusted}">Show details</a>`
    • Untrusted data -> `encodeURIComponent()`
1.1.3.3 Use proven utilities for encoding.
1.1.3.4 DOM-based XSS: Encode both on the server and client side.
1.1.3.5 Add `HTTPOnly`, `Secure` attributes on the Session cookie:
app.use(express.session({
    secret: "s3Cur3",
    key: "sessionId",
    cookie: {
        httpOnly: true,
        secure: true
    }
});
1.1.3.6 Add content security policy header
var policy = {
    defaultPolicy: {
        "default-src": ["'self'"],
        "img-src": ["static.example.com"]
    }
}

helmet.csp.policy(policy);

1.1.4 HTTP parameter pollution

Express populate HTTP request parameters with the same name in an array. 

// POST firstname=John&firstname=Mary
req.body.firstname
// => ["John", "Mary"]

This mechanism can be exploited by an attacker to:

1.1.4.1 trigger type errors on the server (uncaught errors crash it and cause DoS)
1.1.4.2 modify application behavior (e.g. store arrays in the DB)
1.1.4.3 bypass input validation applied on strings in the code, WAF, browser filters
["John", "John"] + "Doe"
// => "John, John Doe"

Countermeasures:

1.1.4.4 Check the expected type as a part of input validation.
1.1.4.5 Incorporate input fuzzing into test suites to discover problems.
1.1.4.6 Implement a robust error handling mechanism, using `try/catch`, `domain` and `cluster`.

1.1.5 Regular Expression Denial of Service (REDoS)

Evil regexps can take exponential execution time when applied to certain non-matching inputs. By default, regexps get executed in the event loop thread, so it can be exploited for DoS attacks.

Evil regexp pattern requiremenets:

1.1.5.1 Grouping with repetition
1.1.5.2 Inside repeated group, repetition (`(a+)+`) or alternation with overlapping (`(a|aa)+`)
1.1.5.3 Example: Input pattern `aaaaaaaaaaaaa!` for a commonly used URL validation regexp
  • 30 input characters take 6sec, 40 take 3hr 46min.

Review the regexps in the code for evil patterns. Do not use user supplied inputs as regexps.

1.1.6 Don't use eval

Indirect forms of `eval`: 

setInterval(String, number);
setTimeout(String, number);
new Function(String);

1.1.7 Avoid command injection

`childprocess.exec` is actually a `bash` interpreter, not a program launcher. Do not pass user input to `childprocess.exec` to avoid potential commands being injected through subshells. Using `childprocess.execFile` is a safe alternative.

1.1.8 Temp files

Instead of creating temporary files (to avoid disk space usage) use Streams.

1.1.9 Cookie theft

Any third-party javascript can read cookies on the page it's being run. To prevent this, set the `HttpOnly` flag on cookies.

1.1.10 Content Security Policy

Introduce a security poilicy for your domain through an HTTP header e.g. `Content-Security-Policy: default-src 'self' *.home.com`.
